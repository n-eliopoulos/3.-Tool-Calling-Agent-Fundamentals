Designing Input Schemas

Schema design directly affects argument accuracy. Required vs. optional, enums for categorical values, nested objects for complex inputs, and description strings on individual properties all guide Claude toward correct arguments.

Schema design patterns

{
  "name": "check_compliance_status",
  "description": "Check the compliance status of a FINRA-registered firm.",
  "input_schema": {
    "type": "object",
    "properties": {
      "firm_id": {
        "type": "string",
        "description": "CRD number of the firm"
      },
      "rule_category": {
        "type": "string",
        "enum": ["trading", "reporting", "communications"],
        "description": "Category of rules to check"
      },
      "date_range": {
        "type": "object",
        "properties": {
          "start": {"type": "string", "description": "ISO date (YYYY-MM-DD)"},
          "end": {"type": "string", "description": "ISO date (YYYY-MM-DD)"}
        },
        "description": "Optional date range to filter results"
      }
    },
    "required": ["firm_id", "rule_category"]
  }
}

Key principles

  • Required vs. optional: Mark parameters as required only when the tool can't function without them. Optional parameters let Claude omit what isn't needed.
  • Enums: Use enum for categorical inputs. Claude will pick from the list instead of guessing free-form values.
  • Nested objects: Group related parameters. Claude handles nested structures well when each property has a description.
  • Property descriptions: Add description to every property — Claude reads these to understand what values to provide.
Try it: Build the check_compliance_status schema in the sandbox and test it with natural language queries like "Check trading compliance for firm 12345 in Q3 2024."
Well-designed schemas constrain Claude's choices to valid values. Enums, required fields, and property descriptions are your primary tools for getting accurate arguments.
