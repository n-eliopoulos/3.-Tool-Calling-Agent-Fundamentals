Tool Schemas: Structure

A tool definition tells Claude what a tool does and what arguments it accepts. The schema uses JSON Schema format, the same standard used in OpenAPI specs.

The three required fields

{
  "name": "get_stock_price",
  "description": "Get the current stock price for a given ticker symbol.",
  "input_schema": {
    "type": "object",
    "properties": {
      "ticker": {
        "type": "string",
        "description": "Stock ticker symbol (e.g., 'AAPL', 'MSFT')"
      }
    },
    "required": ["ticker"]
  }
}
  • name — A unique identifier. Use snake_case. Claude uses this to specify which tool to call.
  • description — A natural language explanation of what the tool does. This is a prompt — Claude reads it to decide when to use the tool.
  • input_schema — JSON Schema defining the arguments. Includes properties, required, and type for each parameter.
Try it: The sandbox below has a pre-loaded get_stock_price tool definition. Send a message with the tool defined and inspect the raw tool_use response block Claude returns.
Tool schemas are contracts between your code and the model. A well-defined schema with clear descriptions leads to accurate tool calls. A vague schema leads to wrong arguments and missed calls.
