Module 1: Tool Calling

Tool Errors and Recovery

Tools fail: network errors, invalid inputs, missing data, timeouts. How you return errors to Claude determines whether it recovers gracefully or produces a bad answer.

Returning structured errors

# Return error as a tool_result with is_error flag
{
    "type": "tool_result",
    "tool_use_id": call.id,
    "content": "Error: Firm CRD#99999 not found in the registry.",
    "is_error": True
}

# Claude reads the error and adapts:
# - Retries with corrected arguments
# - Tries an alternative approach
# - Informs the user gracefully

Common failure modes

  • Invalid arguments: Claude passes a bad value. Return a clear error message — Claude often self-corrects on retry.
  • Not found: The requested data doesn't exist. Claude should inform the user rather than hallucinate.
  • Timeout: The tool takes too long. Return a timeout error so Claude can explain the delay.
  • Rate limiting: Too many calls. Return the error and Claude can suggest waiting or trying fewer queries.
Try it: The sandbox simulates tool failures. Implement error handling that returns structured errors. Watch Claude retry with corrected arguments when a tool returns an error, and inform the user gracefully when data isn't found.
Always return errors as tool_result messages with is_error: true — never swallow them. Claude is remarkably good at recovering from tool failures when it gets clear error messages.
