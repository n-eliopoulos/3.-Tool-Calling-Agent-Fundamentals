Tool Calling & Agent Fundamentals

Module 2: Agent vs. Function

What Makes an Agent?

If any of four capabilities is hardcoded — the goal, the choice of action, the observation of results, or the decision of what to do next — the system is a function, not an agent, regardless of what it's marketed as. The four capabilities that distinguish an agent from a function are listed below.

The four requirements

  1. A goal: The agent knows what it's trying to achieve. "Investigate firm X's compliance posture" — not a hardcoded sequence of steps.
  2. The ability to choose actions: The agent decides which tool to call next based on the current situation, not a predetermined order.
  3. The ability to observe results: The agent reads tool outputs and incorporates them into its reasoning.
  4. The ability to decide what to do next: Based on observations, the agent adapts — it might change strategy, call a different tool, or decide it's done.

The litmus test

QuestionFunctionAgent
Are the steps predetermined?YesNo — chosen at runtime
Does it adapt based on results?No — same path regardlessYes — strategy changes
Can it decide it's done?Runs to completionStops when goal is met
Can it recover from failures?Fixed error handlingTries alternative approaches
Marketing vs. reality: Many systems marketed as "agents" are actually tool-calling functions with retries. That's fine — they work well. But calling them agents sets wrong expectations about their flexibility.
An agent chooses its own actions and adapts based on results. If the steps are hardcoded, it's a function — and that's usually the right choice.
