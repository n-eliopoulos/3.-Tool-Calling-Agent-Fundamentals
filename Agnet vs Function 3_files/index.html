Module 2: Agent vs. Function

Agent Identity: Who Is the Agent Acting As?

Most agents at FINRA are acting on behalf of a specific user. The user authenticated to a front end, that front end forwarded the request, and somewhere in the chain an agent is doing work, but the user's identity, permissions, and accountability flow through everything the agent does. Background and batch agents that run without a user (overnight risk recalculations, scheduled compliance scans) exist, but the user-initiated case is the dominant production pattern. This slide is about that case.

Three questions every user-facing agent must answer

  1. Who is the user? Identity comes from an authenticated session — typically OIDC against FINRA's identity provider (FIP).
  2. What is the user allowed to do? The agent's effective permissions are at most the user's permissions. Active Directory groups (or equivalent attribute-based claims) determine which tools and data are available.
  3. How does identity propagate? When the agent calls a downstream service, that service must also know which user the call is for. The agent cannot just use a service account — that would erase the user's identity and break per-user authorization.

Why a service account is not enough when there is a user

Audit trail"The agent did it""User X did it via the agent"

For genuinely autonomous workloads — overnight batch, scheduled processing, infra automation — a service account is the right call, with audit logs that capture the job context (what was scheduled, by whom, when) instead of a user identity. Don't force an on-behalf-of pattern where there isn't a user to be acting on behalf of.

Forward reference: The mechanics of propagating user identity to downstream services — bearer tokens, scoped service tokens, and OAuth 2.0 token exchange (RFC 8693) — are covered in detail in Security, Governance & Data Pipelines (Track 6 Level 300, module 4: Cross-Service Identity & Token Exchange). For now, hold the mental model: when an agent is acting for a user, that user's identity travels with every call.
The dominant case is on-behalf-of: the agent inherits the user's identity, not its own. Designing a user-facing agent without an answer to "who is this acting as?" creates a system that's impossible to secure, audit, or trust. Genuinely autonomous workloads use service accounts and a different audit model — both patterns are legitimate; pick the right one for the job.
