Module 1: Tool Calling

How Claude Tool Use Works

Claude's tool calling follows a precise request/response cycle. Understanding this cycle is the foundation for everything in this course.

The four-phase cycle

  1. Define tools: You send tool definitions (name, description, input schema) alongside your messages. Claude learns what tools are available.
  2. Claude responds with tool_use: Instead of (or alongside) text, Claude returns a tool_use content block with the tool name and arguments it chose.
  3. You execute and return tool_result: Your code runs the actual function, then sends the result back as a tool_result message.
  4. Claude incorporates the result: Claude reads the tool result and generates its final response, grounded in real data.
# The message flow:
# 1. You send:   tools=[...], messages=[user message]
# 2. Claude returns: content=[{type: "tool_use", name: "get_firm", input: {id: "123"}}]
# 3. You send:   messages=[..., {role: "user", content: [{type: "tool_result", ...}]}]
# 4. Claude returns: content=[{type: "text", text: "Firm 123 is..."}]

Key details

  • Claude decides whether to call a tool — it may answer directly if it has enough information.
  • The tool_use block includes a unique id that you must reference in the tool_result.
  • Claude can return multiple tool_use blocks in a single response (parallel calls).
  • The cycle can repeat — Claude may call another tool after seeing the first result.
Alternative considered: Skip native tool use and instead instruct the LLM in the prompt to output JSON describing the tool call, then parse and execute it yourself. This works often enough to be tempting in early prototypes, but it fails predictably in production: the model wraps the JSON in prose, mixes formats across runs, hallucinates parameter names, and gives you no structured signal for when it's calling a tool versus answering directly. Structured tool use gives you typed inputs validated against your schema and a clear tool_use block separate from the prose response. The prompt-and-parse path is acceptable only when your model genuinely doesn't support native tool use; otherwise structured tool use costs nothing extra and saves you the parser.
Every tool-calling interaction follows the same four-phase cycle: define → tool_use → execute → tool_result. Master this loop and you can build any tool-calling system.
